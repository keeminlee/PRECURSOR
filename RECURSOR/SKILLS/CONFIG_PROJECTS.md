# SKILL: CONFIG_PROJECTS

> **Type:** Reusable task spec
> **Trigger:** One-time setup, or when refreshing the project registry
> **Invocation:** "Run CONFIG_PROJECTS"

---

## Purpose

Create or refresh `RECURSOR/PROJECTS.md` — the centralized inventory of all projects and subrepos in the workspace. This is the registry that `@Start` and other skills read to discover what exists, what is active, and which files to hydrate for each project.

If `RECURSOR/PROJECTS.md` already exists, run as a **diff** — add newly discovered projects, flag stale entries, update key file paths — do not overwrite the whole file.

---

## Phase A — Scan (2-level depth)

List the workspace root. Then, for each top-level folder that is **not** infrastructure, list its direct children.

**Exclude these infrastructure folders entirely:**
- `CANON/`
- `docs/`
- `PLANS/`
- `.github/`
- `.obsidian/`
- `.venv/`
- `_backup/`
- `data/`

For each candidate folder found (top-level or one level deep inside a top-level), check for the presence of these indicator files:

| Indicator | Signals |
|---|---|
| `.git/` | Distinct repo boundary |
| `README.md` | Project-level documentation exists |
| `CURRENT_PRIORITIES.md` or `*PRIORITIES*` | Active work in progress |
| `*IMPLEMENTATION_PROGRESS*` or `*PROGRESS*` | Active implementation log |
| `main.py` or `main_graph.py` | Python entry point |
| `package.json` | Node/frontend entry point |
| `pyproject.toml` or `requirements.txt` | Python project |
| `main.tf` | Terraform IaC project |
| `azure-pipelines.yml` or `.github/workflows/` | CI/CD wired up |

---

## Phase B — Classify

Git-tracking is the primary signal — a folder with `.git/` is a real project boundary. Weight it heavily.

Assign each discovered folder a status using this priority order:

| Status | Criteria |
|---|---|
| `ACTIVE` | **Has `.git/`** AND (has a priorities file OR has CI config OR was modified within the last 60 days). Git-tracking is required for ACTIVE — a non-git folder cannot be ACTIVE. |
| `STUB` | Has `.git/` but does not meet ACTIVE criteria (no priorities file, no CI, modified >60 days ago, or clearly ancillary/superseded). Also: non-git folders with structured code content that aren't clearly archived. |
| `ARCHIVED` | A sibling `.zip` exists with the same name, OR no `.git/` and last modified >90 days ago, OR clearly superseded by a newer version in the same parent with `.git/`. |
| `DATA` | Contains primarily documents, PDFs, or demo artifacts — not code. No `.git/`. |

For ambiguous cases, prefer `STUB` over `ARCHIVED`.

**ACTIVE project ordering:** When writing the inventory table and detail blocks, list ACTIVE projects first, ordered by estimated hotness descending:
1. Has priorities/progress files AND git AND recent modification → hottest
2. Has git AND CI AND recent modification (no priorities file)
3. Has git AND recent modification only

Non-ACTIVE projects follow after, grouped by status (STUB, then ARCHIVED, then DATA).

## Phase C — Write `RECURSOR/PROJECTS.md`

Write the file using this exact schema:

```markdown
# PROJECTS — Project Registry

> Auto-generated by CONFIG_PROJECTS skill. Last updated: {date}.
> Verify the project list and key file paths before trusting @Start rehydration.

---

## Project Inventory

| ID | Name | Path | Status | Description |
|---|---|---|---|---|
| {id} | {name} | {path} | ACTIVE/STUB/ARCHIVED/DATA | {one-line description} |

---

## Project Details

### {ID}: {Name}
> **Path:** `{path}`
> **Status:** `{status}`
> **Auto-detected:** {yes/no}

| Slot | Path | Notes |
|---|---|---|
| README | {path or MISSING} | |
| Priority stack | {path or MISSING} | |
| Implementation log | {path or MISSING} | |
| Entry point(s) | {path(s) or MISSING} | comma-separated if multiple |
| Frontend | {path or MISSING} | |
| CI config | {path or MISSING} | |

---
```

Repeat the `### {ID}: {Name}` block for every project in the inventory table.

Leave slots as `MISSING` where the file was not found during the scan — do not guess or invent paths.

---

## Phase D — Verify Prompt

After writing `RECURSOR/PROJECTS.md`, output the following checklist to the user in chat (do not write it to a file):

```
PROJECTS.md written. Please verify before @Start uses it:

[ ] Project list is complete — no active projects are missing
[ ] Statuses are correct (ACTIVE / STUB / ARCHIVED / DATA)
[ ] Key file paths are accurate — check any slots marked MISSING
[ ] No infrastructure folders accidentally included

To correct anything, edit RECURSOR/PROJECTS.md directly or re-run CONFIG_PROJECTS.
```

Do not mark this skill complete until the user has reviewed and confirmed (or corrected) the output.

---

## Completion Criteria

- [ ] `RECURSOR/PROJECTS.md` written with Project Inventory table and one detail block per project
- [ ] All discovered projects have a status assigned
- [ ] Key file slots populated where detectable; `MISSING` where not
- [ ] Verify prompt delivered to user in chat
