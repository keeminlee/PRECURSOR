# SKILL: CONFIG_PROJECTS

> **Type:** Reusable task spec
> **Trigger:** First-time setup (via BOOTSTRAP), or when refreshing the project registry
> **Invocation:** "Run CONFIG_PROJECTS"

---

## Purpose

Create or refresh `PROJECTS.md` — the centralized inventory of all projects in the workspace. This is the registry that @PrecursorStart reads to discover what exists and what is active.

If `PROJECTS.md` already exists, run as a **diff** — add newly discovered projects, flag stale entries, update key file paths — do not overwrite the whole file.

---

## Phase A — Scan (2-level depth)

List the workspace root. Then, for each top-level folder that is **not** infrastructure, list its direct children.

**Exclude these infrastructure folders entirely:**
- `SKILLS/`
- `.github/`
- `PLANS/`
- `demo/`
- `docs/`
- `.venv/`
- `node_modules/`

For each candidate folder found, check for these indicator files:

| Indicator | Signals |
|---|---|
| `.git/` | Distinct repo boundary |
| `README.md` | Project-level documentation exists |
| `package.json` | Node/frontend entry point |
| `pyproject.toml` or `requirements.txt` | Python project |
| `main.py` or `index.js` or `index.ts` | Entry point |
| `.github/workflows/` | CI/CD wired up |

---

## Phase B — Classify

Assign each discovered folder a status:

| Status | Criteria |
|---|---|
| `ACTIVE` | Has `.git/` AND shows signs of current work (recent commits, priorities file, CI config) |
| `STUB` | Has `.git/` but no recent activity, or a non-git folder with structured code |
| `ARCHIVED` | A sibling `.zip` exists, or clearly superseded, or no activity in 90+ days |
| `DATA` | Contains primarily documents or data — not code |

For ambiguous cases, prefer `STUB` over `ARCHIVED`.

---

## Phase C — Write `PROJECTS.md`

Write the file at the workspace root using this schema:

```markdown
# PROJECTS — Project Registry

> Auto-generated by CONFIG_PROJECTS skill. Last updated: {date}.
> Verify the project list and key file paths before trusting rehydration.

---

## Project Inventory

| ID | Name | Path | Status | Description |
|---|---|---|---|---|
| {id} | {name} | {path} | ACTIVE/STUB/ARCHIVED/DATA | {one-line description} |

---

## Project Details

### {ID}: {Name}
> **Path:** `{path}`
> **Status:** `{status}`

| Slot | Path | Notes |
|---|---|---|
| README | {path or MISSING} | |
| Entry point(s) | {path(s) or MISSING} | |
| CI config | {path or MISSING} | |

---
```

Repeat the detail block for every project in the inventory table.

---

## Phase D — Verify Prompt

After writing `PROJECTS.md`, output this checklist to the user in chat:

```
PROJECTS.md written. Please verify:

[ ] Project list is complete — no active projects are missing
[ ] Statuses are correct (ACTIVE / STUB / ARCHIVED / DATA)
[ ] Key file paths are accurate
[ ] No infrastructure folders accidentally included

To correct anything, edit PROJECTS.md directly or re-run CONFIG_PROJECTS.
```

---

- [ ] `PROJECTS.md` written with inventory table and detail blocks
- [ ] All discovered projects have a status assigned
- [ ] Verify prompt delivered to user
